# 모델·에포트 라우팅 (Model & Effort Routing)

## 언어 정책 (필수)

- 내부 분석, 계획, 문제 해결, 도구 호출, 검색어 작성 등 모든 사고 및 작업 과정은 **영어를 우선**한다.
- 사용자에게 전달하는 진행 업데이트와 최종 답변은 **한국어를 우선**한다.
- 코드, 명령어, API, 라이브러리, 모델명, 파일명과 기타 기술 용어는 정확성과 업계 관례를 위해 영어로 작성해도 된다.
- 사용자가 특정 언어를 명시하면 해당 요청을 우선한다.

## RTK/Caveman 상시 적용 (필수)

- 런타임에 `caveman` skill이 설치되어 있으면, 매 세션의 첫 응답부터 별도 호출 없이 반드시 Caveman `full` 응답 형식을 적용한다.
- 적용 상태는 모든 턴과 세션에서 유지한다. 사용자가 `stop caveman` 또는 `normal mode`를 명시한 경우에만 해제한다.
- 진행 업데이트와 최종 답변 모두에 적용한다. 불필요한 인사, 도구 호출 서술, 장식 표, 장황한 로그를 제거한다.
- 사용자의 주 언어를 유지하고 코드, 명령어, API, 파일 경로, 정확한 오류 문자열은 변경하지 않는다.
- 보안 경고, 비가역 작업 확인, 압축으로 의미가 불명확해지는 절차만 명료성을 위해 일시적으로 확장하고, 이후 즉시 Caveman 형식으로 복귀한다.

## 목적과 기본 원칙

- 총비용과 지연은 대략 `모델 등급 × 에포트 × 에이전트 수`에 비례한다.
- 세 다이얼을 한꺼번에 올리지 말고, 필요한 다이얼만 한 단계씩 올린다.
- 기본값은 **단일 에이전트 · GPT-5.6-Terra · high**이다. (첫 표 기준 주작업 가성비 종결 구간)
- **★ Max 에포트 영구 금지:** 모든 모델 라인업에서 `max` 에포트는 가성비가 박살 나거나 심각한 지능 저하(능지 하락)를 유발하므로 **상시 사용을 전면 금지**한다.
- 시스템·플랫폼·사용자 지침과 더 가까운 경로의 `AGENTS.md`가 이 문서보다 우선한다.

## 모델 라인업

| 모델 | 위치 | 주 용도 |
|---|---|---|
| `gpt-5.6-sol` (GPT-5.6-Sol) | 프론티어 | 대형 설계, 최고 수준의 똑똑함이 필요한 계획, 핵심 난제 해결 |
| `gpt-5.6-terra` (GPT-5.6-Terra) | 밸런스 | 일반 구현, 기능 개발, 실질적인 메인 코딩 및 주작업 (가성비 최고) |
| `gpt-5.6-luna` (GPT-5.6-Luna) | 고속·저비용 | 기계적 조사, 정보 검색, 단순 탐색 |

### Sol

- 가장 똑똑한 판단이 필요한 아키텍처 설계 및 계획 수립 단계에 사용한다.
- `sol-high`와 `sol-xhigh`는 성능 차이가 거의 느껴지지 않으므로, 비용 효율을 위해 가급적 `sol-high`를 우선 선택한다.
- `sol-max`는 가성비가 완전히 박살 났으므로 절대 사용하지 않으며, 더 높은 추론이 필요할 때는 차라리 `xhigh`를 유지하거나 필요시 `ultra`로 전환한다.

### Terra

- 일반적인 기능 구현, 버그 수정, 중간 규모 리팩터링 등 실질적인 모든 '주작업'의 메인 모델이다.
- 주작업용으로 가성비가 매우 우수하며, 특히 출력량이 많을 때는 `terra-xhigh`가 `sol-medium`과 지능이 동급이면서 가성비가 매우 유리하므로 이를 우선 고려한다.
- `terra-max`는 가성비가 파탄 났으므로 쓰지 않는다. 이 비용을 쓸 바엔 `sol-high`를 쓰는 것이 낫다.

### Luna

- 파일 검색, 구조 조사, 정보 탐색 단계에서 `luna-high`를 사용 시 최적의 가성비를 보여준다.
- `luna-medium`은 효용성이 낮아 쓸 일이 없으며, `luna-xhigh`를 쓸 바엔 성능과 밸런스가 좋은 `sol-medium`을 쓴다.
- `luna-max`는 심각한 지능 저하 및 비용 낭비(그돈씨)가 발생하므로 절대 금지한다.

## 에포트 레벨

| 에포트 | 사용 기준 |
|---|---|
| low | 단순 검색, 분류, 기계적 작업 |
| medium | 가벼운 작업 및 요약 (`sol-medium`은 `terra-xhigh`와 동급 성능) |
| high | 일반적인 구현 및 계획 수립의 **기본값** (가성비 최적화 구간) |
| xhigh | 깊은 디버깅, 대형 리팩터링, 가장 똑똑한 지능이 필요할 때 |
| **max** | **[영구 사용 금지]** 비용 폭망 및 지능 저하 구간. 절대 진입하지 않는다. |
| ultra | 최대 추론과 자동 작업 위임이 모두 필요한 제한적 작업 |

- Luna는 low, high를 주로 사용한다.
- Sol과 Terra는 low, medium, high, xhigh, ultra를 사용할 수 있다.
- 상향 순서는 `현재 모델에서 effort 한 단계 상승 → 문제 분해 → 독립 조사 병렬화 → Luna에서 Terra로 또는 Terra에서 Sol로 모델 상승 → xhigh 유지 → 필요 시 ultra`이다. (**max 단계는 완전히 배제**)

## 복잡도 티어

| 티어 | 기준 |
|---|---|
| S | 단일 파일, 익숙한 패턴, 작은 버그 수정, 단순 정리 |
| M | 기능 단위, 설계가 비교적 명확함, 여러 파일 수정 |
| L | 크로스스택, 재설계, 대형 리팩터링, 감사 |
| XL | 신규 도메인, 탐색적 설계, 불확실성과 직렬 임계경로가 큼 |

파일 수만으로 티어를 정하지 않는다. 요구사항 불확실성, 파급 범위, 데이터·보안 위험, 되돌리기 비용과 검증 난도를 함께 고려한다.

## 티어 × 단계 매트릭스

`●●●`는 멀티에이전트가 정당할 수 있음을 뜻한다. 상위 지침이나 사용자가 허용한 경우에만 사용한다.

> **※ 라우팅 원칙:** 첫 표의 평가 데이터에 기반하여, 가성비가 파탄 난 `max` 에포트 호출을 전면 제거하고 실질적 동급 효율 모델로 우회 라우팅한다.

| 티어 | ① 계획 | ② 계획 검토 | ③ 구현 | ④ PR 리뷰 |
|---|---|---|---|---|
| S | 생략 또는 Luna·high | 셀프 검토 | Luna·high | Terra·high |
| M | Sol·medium 또는 Terra·high | Terra·high | Terra·high *(출력 많으면 xhigh)* | Terra·high |
| L | Sol·high | Sol·xhigh ●●● | Terra·high 또는 Sol·medium | Sol·high |
| XL | Sol·xhigh | Sol·xhigh ●●● | Terra·xhigh *(핵심만 Sol·high)* | Sol·xhigh |

예외:

- S 작업이라도 보안, 결제, 권한, 데이터 손실 위험이 있으면 Terra·high 이상을 사용한다.
- 기존 최난도 단발성(max 요구) 문제 직면 시, `sol-max`로 올리지 말고 `sol-xhigh`를 유지하거나 필요시 `ultra`로 즉시 우회한다.
- 모델이나 effort를 높였다는 이유로 테스트와 검증을 생략하지 않는다.

## 메인 세션 체크 (필수)

새 작업을 받으면 첫 진행 응답에서 다음을 짧게 명시한다.

1. 복잡도 티어: S / M / L / XL
2. 단계: 계획 / 계획 검토 / 구현 / PR 리뷰
3. 권장 모델과 effort
4. 단일 또는 멀티에이전트 여부
5. 현재 세션 설정이 권장값과 다르면 전환 안내

예시:

> 판정: M · 구현 단계. Terra·high와 단일 에이전트가 권장됩니다. (현재 설정이 과하여 /reasoning 하향을 권고합니다.)

- 현재 설정이 과하면 `/model`과 `/reasoning`으로 하향 전환을 권고한다.
- 현재 설정이 부족하면 작업 시작 전에 `/model`과 `/reasoning`으로 상향 전환을 권고한다.
- 에이전트는 메인 세션 설정을 스스로 변경했다고 주장하지 않는다. 안내만 한다.
- 사용자가 전환하지 않고 계속 진행하면 현재 설정으로 작업한다.

## 모델 고정 exec 라우팅 (필수)

- native `spawn_agent`, native fan-out, 모델명을 프롬프트에만 적는 위임은 사용하지 않는다.
- native 작업자는 부모의 모델과 effort를 상속할 수 있으므로 비용 라우팅 수단으로 간주하지 않는다.
- 모든 보조 작업자는 별도 `codex exec` 프로세스로 실행하며 `-m`과 `-c model_reasoning_effort=...`를 반드시 명시한다.
- 실행 위치를 고정하기 위해 `-C <repo-root>`를 반드시 명시한다.
- 조사·검토 작업자는 `-s read-only`, 구현 작업자는 필요한 경우에만 `-s workspace-write`를 사용한다.
- 감사 추적이 필요한 작업에는 `--ephemeral`을 사용하지 않는다.
- 결과 수집은 `--json` 또는 `-o <output-file>`을 사용한다.
- 기존 native fan-out 결과는 운영 원칙 위반 결과로 격리하고, 의사결정에 사용하기 전에 모델 고정 exec로 단독 재검증한다.

기본 실행 형식:

```bash
rtk codex exec \
  -C "<repo-root>" \
  -m <model> \
  -c 'model_reasoning_effort="<effort>"' \
  -s <read-only|workspace-write> \
  -o <output-file> \
  "<bounded-task>"
```

## exec 작업자 라우팅

작업자는 하위 작업이 실제로 독립적이고, 분리 실행의 이득이 조정 비용보다 클 때만 사용한다.

### 기계적 조사

- 예: 파일 존재 확인, `rg` 검색, 참조 목록화, 테스트 실패 분류, 반복 패턴 조사.
- 고정 설정: `-m gpt-5.6-luna -c 'model_reasoning_effort="high"'`
- 기본 sandbox: `-s read-only`
- 출력 범위를 좁게 지정하고 최종 판단은 메인 에이전트가 수행한다.

### 일반 구현

- 예: 독립 패키지 수정, 서로 겹치지 않는 파일 작업, 분리된 테스트 작성.
- 고정 설정: `-m gpt-5.6-terra -c 'model_reasoning_effort="high"'`
- 대량 출력 또는 복잡한 구현만 `model_reasoning_effort="xhigh"`로 올린다.
- 구현이 필요할 때만 `-s workspace-write`를 사용한다.
- 동일 파일을 여러 exec 작업자가 동시에 수정하지 않게 한다.

### 판단·비평

- 예: 근본 원인 판정, 아키텍처 비평, 적대적 리뷰, 보안·성능·접근성 렌즈.
- 균형 설정: `-m gpt-5.6-sol -c 'model_reasoning_effort="medium"'`
- 고난도 설정: `-m gpt-5.6-sol -c 'model_reasoning_effort="high"'`
- 기본 sandbox: `-s read-only`
- 근거와 불확실성을 보고하게 하고 최종 종합은 메인 에이전트가 수행한다.

## 멀티프로세스 작업자 규칙

- native 멀티에이전트 기능은 사용하지 않는다.
- 하위 작업이 서로 독립적이고, 병렬화로 실제 시간이 줄며, 결과를 명확히 병합할 수 있을 때만 모델 고정 exec 작업자를 사용한다.
- 사용자가 요청했거나 상위 지침이 허용한 경우에만 병렬 실행한다.
- 저장소 전수 감사, 독립 리뷰 렌즈, 문서↔코드 검증, 여러 패키지의 독립 조사에 적합하다.
- 단일 파일 수정, 순차 의존 작업, 작은 버그 수정, 동일 문제 중복 할당에는 사용하지 않는다.
- 동시 활성 exec 작업자는 환경 제한을 따르며, 별도 제한이 없다면 최대 4개로 제한한다.
- 각 구현 작업자의 파일 소유권을 미리 분리한다.
- 모든 작업자 종료 후 메인 에이전트가 결과, 변경 충돌, 테스트 상태를 직접 검증한다.

## 실전 요약

| 작업 | 권장 설정 | 비고 (첫 표 데이터 기준) |
|---|---|---|
| 파일 검색, 목록화, 탐색 조사 | **Luna·high** | 탐색/조사 시 가성비 가장 우수 |
| 가벼운 요약 및 정리 작업 | **Luna·high** | `luna-medium`은 효용성 없음 |
| 오타, 포맷, 단순 설정 변경 | **Luna·high** 또는 **Terra·high** | 체급 안정성 고려 |
| 일반 기능 구현 (주작업 기본값) | **Terra·high** | 주작업 가성비 우수 종결 구간 |
| 대량 출력이 동반되는 주작업 | **Terra·xhigh** | 출력 많을 시 가성비 매우 유리 |
| M급 계획 및 균형 잡힌 분석 | **Sol·medium** | `terra-xhigh`와 동급 성능 효율 |
| L급 설계 및 고지능 계획 수립 | **Sol·xhigh** | `sol-high`와 큰 차이 없는 종결 구간 |
| 어려운 근본 원인 분석 / 최난도 분석 | **Sol·xhigh** | 가장 똑똑한 성능이 필요할 때 추천 |
| **최난도 단발 문제 (기존 max 구간)** | **Sol·xhigh** 또는 **ultra** | **모든 `max` 세팅 절대 금지** |

## 금지 패턴

- **모든 모델에서 `max` 에포트 세팅을 절대 시작하거나 사용하지 않는다.** (`sol-max` = 가성비 박살, `terra-max` = sol-high 미만 잡, `luna-max` = 그돈씨 및 능지하락)
- 효율이 떨어지는 부적절한 세팅인 `luna-medium`, `luna-xhigh`를 사용하지 않는다. (각각 `luna-high`와 `sol-medium`으로 대체)
- 단순 검색 및 탐색 조사를 고비용 모델(Sol)에 맡기지 않는다.
- 복잡한 최종 판정을 Luna 결과만으로 확정하지 않는다.
- `ultra`를 일반적인 고성능 모드처럼 상시 사용하지 않는다.
